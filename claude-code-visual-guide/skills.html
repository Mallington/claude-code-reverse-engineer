Chapter 6 — The Skills System

The Skills System

On-demand behavioral modules that guide complex workflows — loaded lazily to keep token costs near zero until the moment they are needed.

Bootstrap Mechanism

The SessionStart Hook Flow

How using-superpowers gets injected at the start of every session — step by step.

1
Session starts, hooks.json consulted
Claude Code reads .claude/hooks.json and finds a SessionStart hook registered by the Superpowers plugin.
{ "SessionStart": ["run-hook.cmd session-start"] }
2
run-hook.cmd session-start executes
The hook runner shell script is invoked with the session-start argument, triggering the bootstrap sequence.
3
Script reads using-superpowers/SKILL.md
The session-start script locates and reads the bootstrap skill file from the plugin's skills directory.
~/.claude/plugins/cache/<marketplace>/superpowers/<version>/skills/using-superpowers/SKILL.md
4
Content wrapped in <EXTREMELY_IMPORTANT>
The skill content is wrapped in priority tags to ensure the model treats it as high-priority instruction.
<EXTREMELY_IMPORTANT> [skill content — ~1,200 tokens] </EXTREMELY_IMPORTANT>
5
Returns as hookSpecificOutput.additionalContext
The hook outputs a JSON object. Claude Code reads the additionalContext field from hookSpecificOutput.
{ "hookSpecificOutput": { "additionalContext": "..." } }
6
Claude Code wraps in <system-reminder>
Claude Code takes the additional context and wraps it in a <system-reminder> block for injection.
<system-reminder> <EXTREMELY_IMPORTANT>...</EXTREMELY_IMPORTANT> </system-reminder>
7
Injected as first block of first user message
The system-reminder is prepended to the human turn of the first message in the conversation, before any user text.
8
Model reads skill, learns about the skill system
The model sees the bootstrap skill content, understands what skills are available, and knows to invoke the Skill tool when appropriate tasks arise. The catalog is already in context — the model can match tasks to skill names immediately.

Critical Concepts

Key Insights

Description = Trigger Only
The description field in a skill's YAML frontmatter must contain ONLY triggering conditions — never a workflow summary. When a description summarizes the workflow, the model follows the description instead of loading and reading the full skill content.
BAD: "Guides ideation: explores problem, generates options, evaluates trade-offs"
GOOD: "Before any creative work — features, components, system design"
Lazy Loading
Skills cost 0 tokens until invoked. Only the skill catalog (~600 tokens of names and descriptions) is always present. Individual skill content — 500 to 3,000 tokens each — enters context only when the model calls the Skill tool. A session with no complex tasks carries no skill overhead.
Chain Awareness
Skills reference each other through REQUIRED SUB-SKILL markers in their content. subagent-driven-development tells the model to invoke dispatching-parallel-agents and test-driven-development. This creates a composable pipeline where complex workflows self-assemble from small focused skill files.
