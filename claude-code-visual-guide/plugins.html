Section 3

          
SessionStart Hook — Deep Dive

          
The session-start script performs five steps every time a session begins. Click any step to see the real code.

        


        


          
          

            
            

              

                
Checks whether ~/.config/superpowers/skills exists — the legacy pre-plugin path. If found, outputs a deprecation warning so the user knows to migrate.

                

                  
session-start (bash)

                  
# Check for legacy skills directory
if [ -d "${HOME}/.config/superpowers/skills" ]; then
  echo "WARNING: Legacy skills dir detected at ~/.config/superpowers/skills" >&2
  echo "Please migrate to the plugin system." >&2
fi

                

              

            

          


          
          

            
            

              

                
Reads the using-superpowers/SKILL.md file from the plugin's skills directory. This is the bootstrapping skill that tells Claude how to use all other skills in the plugin.

                

                  
session-start (bash)

                  
# PLUGIN_ROOT is set by Claude Code to the plugin's cache dir
SKILL_FILE="${CLAUDE_PLUGIN_ROOT}/skills/using-superpowers/SKILL.md"

if [ ! -f "${SKILL_FILE}" ]; then
  echo "ERROR: Skill file not found: ${SKILL_FILE}" >&2
  exit 1
fi

skill_content=$(cat "${SKILL_FILE}")

                

              

            

          


          
          

            
            

              

                
The skill file is arbitrary Markdown. Before embedding it inside a JSON string it must be escaped: backslashes doubled, double-quotes escaped, and newlines replaced with \n.

                

                  
session-start (bash)

                  
escape_for_json() {
  local input="$1"
  local output

  # 1. Escape backslashes first (must be first!)
  output="${input//\\/\\\\}"

  # 2. Escape double quotes
  output="${output//\"/\\\"}"

  # 3. Replace literal newlines with \n
  output="${output//$'\n'/\\n}"

  printf '%s' "$output"
}

session_context=$(escape_for_json "${skill_content}")

                

              

            

          


          
          

            
            

              

                
The escaped content is wrapped in <EXTREMELY_IMPORTANT> XML tags. This signals to Claude that the injected context should be treated as high-priority system instructions.

                

                  
session-start (bash)

                  
context_wrapped="<EXTREMELY_IMPORTANT>\\n${session_context}\\n</EXTREMELY_IMPORTANT>"

# Re-escape the wrapper (the tags themselves are safe,
# but session_context was already escaped above)
final_context="${context_wrapped}"

                

              

            

          


          
          

            
            

              

                
The output format differs by platform. Claude Code (native) uses hookSpecificOutput; Cursor uses additional_context. The script detects the platform via environment variables.

                

                  
session-start (bash)

                  
if [ -n "${CURSOR_PLUGIN_ROOT:-}" ]; then
  # Cursor format
  printf '{"additional_context": "%s"}\n' "$final_context"

elif [ -n "${CLAUDE_PLUGIN_ROOT:-}" ]; then
  # Claude Code native format
  printf '{"hookSpecificOutput": {"hookEventName": "SessionStart", "additionalContext": "%s"}}\n' \
    "$final_context"

else
  # Unknown platform — output nothing
  exit 0
fi

The Polyglot Runner

          

            run-hook.cmd is a single file that executes correctly on both Windows (as a .cmd batch file) and Unix (as a shell script), using the polyglot comment trick.
          

          

            

              run-hook.cmd
            

            
: << 'CMDBLOCK'
@echo off
REM Windows: finds bash.exe on PATH and runs the named script
SET SCRIPT_NAME=%1
FOR /F "delims=" %%i IN ('where bash 2^>nul') DO SET BASH=%%i
"%BASH%" "%~dp0%SCRIPT_NAME%"
goto :eof
CMDBLOCK
# Unix: the CMDBLOCK heredoc was a no-op; run the script directly
SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"
SCRIPT_NAME="$1"
exec bash "${SCRIPT_DIR}/${SCRIPT_NAME}" "$@"

          

        


      

    


    
    

      


        

          
Section 5

          
What a Plugin Can Add

          
A plugin package can contribute four types of components, each living in its own subdirectory and registered automatically at startup.

        


        


          
          

            

              
            

            
Hooks

            
Lifecycle callbacks fired at session start, before/after tool execution, and on prompt submission. Can inject context or block actions.

            
hooks/hooks.json

          


          
          

            

              
            

            
Skills

            
Behavioral workflow modules stored as Markdown files. Loaded on demand via the Skill tool — zero token cost until needed.

            
skills/<skill-name>/

          


          
          

            

              
            

            
Agents

            
Custom agent type definitions with name, description, model, and system prompt. Registered as sub-agents available to the Agent tool.

            
agents/<name>.md

          


          
          

            

              
            

            
Commands

            
Slash commands that appear in the Claude Code command palette. Defined in Markdown with a YAML front-matter description.

            
commands/<name>.md

                json

  "hooks": {
    "SessionStart": [
      {
        "matcher": "startup|clear|compact",

          {

      {
        "matcher": ".*",
        "hooks": [
          {
            "type": "command",
            "command": "\"${CLAUDE_PLUGIN_ROOT}/hooks/run-hook.cmd\" pre-tool-use",
            "async": true
          }
        ]
      }
    ]
  }
}

Four Execution Strategies

          
Every hook declaration has a type field. Four public types determine how the engine executes the hook.


          


            
            

              
💻

              
type: "command"

              
Shell Execution

              
Default type. Spawns a child process via child_process.spawn. Hook input JSON is piped on stdin; stdout lines are parsed for JSON responses.

              

                
Execution flow

                
    
                  
  1. 
                    1
                    Input JSON serialised and piped to process stdin
                  
    2. 
                  
  2. 
                    2
                    Env vars set: CLAUDE_SESSION_ID, CLAUDE_CWD, CLAUDE_PLUGIN_ROOT
                  
    3. 
                  
  3. 
                    3
                    stdout lines parsed as JSON (one object per line)
                  
    4. 
                  
  4. 
                    4
                    Exit code interpreted: 0 = silent ok, 2 = block/continue, other = show to user
                  
    5. 
                

              

              

                
Default timeout: 10 minutes (TOOL_HOOK_EXECUTION_TIMEOUT_MS)

                
Supports bash, powershell, or custom shell

                
Async mode: first line {"async":true} backgrounds the process

                
Interactive prompts via PromptRequest JSON to stdout

              

            


            
            

              
🌐

              
type: "http"

              
HTTP POST

              
Fires an HTTP POST to an external policy server. Hook input JSON is the request body. Response body is parsed as JSON using the same schema as shell hooks.

              

                
Execution flow

                
    
                  
  1. 
                    1
                    SSRF guard validates the resolved IP before connection
                  
    2. 
                  
  2. 
                    2
                    POST body = hook input JSON, Content-Type: application/json
                  
    3. 
                  
  3. 
                    3
                    Header $VAR patterns resolved from allowedEnvVars list only
                  
    4. 
                  
  4. 
                    4
                    Response body parsed as hookJSONOutputSchema
                  
    5. 
                

              

              

                
SSRF guard blocks private/link-local ranges (loopback allowed)

                
Admin allowedHttpHookUrls glob-list enforced

                
Sandbox mode routes through network proxy

                
Env vars only interpolated if explicitly listed

              

            


            
            

              
🤖

              
type: "agent"

              
Agent (LLM Query)

              
Spawns a multi-turn LLM conversation to evaluate a condition. The agent has access to all tools and can inspect the codebase before returning a structured verdict.

              

                
Execution flow

                
    
                  
  1. 
                    1
                    $ARGUMENTS in prompt replaced with hook JSON input
                  
    2. 
                  
  2. 
                    2
                    System prompt injected: agent must verify a stop condition
                  
    3. 
                  
  3. 
                    3
                    Agent runs up to 50 turns using available tools (no subagents)
                  
    4. 
                  
  4. 
                    4
                    Must call SyntheticOutputTool with {ok:true} or {ok:false,reason}
                  
    5. 
                

              

              

                
Default model: small/fast (Haiku-class)

                
Max 50 turns; thinking disabled; non-interactive mode

                
Gets transcript file access for full context

                
ok:false → preventContinuation:true

              

            


            
            

              
💬

              
type: "prompt"

              
Prompt (LLM Query)

              
Single-shot LLM query with no tool use. Simpler and faster than Agent hooks. The model returns {ok, reason} as structured JSON output.

              

                
Execution flow

                
    
                  
  1. 
                    1
                    $ARGUMENTS substituted with hook JSON input in prompt
                  
    2. 
                  
  2. 
                    2
                    Single call via queryModelWithoutStreaming
                  
    3. 
                  
  3. 
                    3
                    Structured JSON output enforced: {ok: boolean, reason?: string}
                  
    4. 
                  
  4. 
                    4
                    If ok:false → blocking error with preventContinuation:true
                  
    5. 
                

              

              

                
Default timeout: 30 seconds

                
Default model: small/fast (Haiku-class)

                
No tool use — faster and cheaper than Agent type

                
Best for simple pass/fail conditions on content

Hook JSON Output Protocol

          
Regardless of execution strategy, all hooks communicate back via hookJSONOutputSchema (from src/types/hooks.ts). Click any highlighted field to see details.


          

            

              hookJSONOutputSchema — sync response
              Click a field to explore
            

            

              

{
  "continue": true,
  "suppressOutput": false,
  "stopReason": "...",
  "decision": "approve",
  "reason": "...",
  "systemMessage": "...",
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    // event-specific fields...
  }
}
              


              

                

                  
                  
Click any highlighted field in the schema to see its type, purpose, and accepted values.

                


                

                  
"continue"

                  
boolean?

                  
Controls whether Claude continues after the hook runs. Defaults to true. Set to false to stop the current operation immediately and display stopReason to the user.

                  

                    
Values

                    

                      true
                      Default. Claude continues normally after the hook.
                    

                    

                      false
                      Stop current operation. Display stopReason message to the user.
                    

                  

                


                

                  
"suppressOutput"

                  
boolean?

                  
When true, the hook's stdout is hidden from the conversation transcript. Useful for hooks that produce verbose diagnostic output that shouldn't pollute context.

                  

                    
Values

                    

                      false
                      Default. stdout is visible in transcript.
                    

                    

                      true
                      stdout is suppressed from transcript and context.
                    

                  

                


                

                  
"stopReason"

                  
string?

                  
Human-readable message displayed to the user when continue: false is returned. Explains why the hook stopped Claude's response.

                  

                    
Example

                    

                      "stopReason"
                      "Blocked: destructive git command detected. Please confirm manually."
                    

                  

                


                

                  
"decision"

                  
"approve" | "block"

                  
Simple approve/block decision for hooks that gate an action. approve allows the action to proceed; block prevents it. Pair with reason for user-visible explanation.

                  

                    
Values

                    

                      "approve"
                      Allow the gated action to proceed.
                    

                    

                      "block"
                      Prevent the action. reason is shown to the user.
                    

                  

                


                

                  
"reason"

                  
string?

                  
Explanation for the decision (approve or block). Shown to the user in the permission dialog or error output. Also used as the reason text in PreToolUse permission decisions.

                  

                    
Example

                    

                      "reason"
                      "This command modifies production database. Manual review required."
                    

                  

                


                

                  
"systemMessage"

                  
string?

                  
A warning or informational message displayed directly to the user in the UI (not injected into Claude's context). Used for out-of-band notifications that don't need to influence the model.

                  

                    
Example

                    

                      "systemMessage"
                      "Warning: 3 uncommitted changes detected. Consider committing before continuing."
                    

                  

                


                

                  
"hookSpecificOutput"

                  
object?

                  
Event-specific return data. Discriminated union on hookEventName. Contains fields that only make sense for a particular event — e.g., permissionDecision for PreToolUse, additionalContext for SessionStart.

                  

                    
Key event-specific fields

                    

                      PreToolUse
                      permissionDecision (allow/deny/ask/passthrough), updatedInput to rewrite tool args
                    

                    

                      PostToolUse
                      additionalContext, updatedMCPToolOutput to rewrite MCP results
                    

                    

                      SessionStart
                      additionalContext, initialUserMessage, watchPaths[]
                    

                    

                      PermissionRequest
                      decision: {behavior:'allow'|'deny', updatedInput?, message?}
                    

                    

                      CwdChanged / FileChanged
                      watchPaths[] to dynamically update the file watcher
                    

                  

                


              

            

          

        


        
        


          
7d

          
SSRF Protection

          
HTTP hooks are guarded at DNS resolution level — the IP is validated after DNS lookup, eliminating the rebind window. Source: src/utils/hooks/ssrfGuard.ts.


          

            


              
              

                

                  
                  Blocked Address Ranges
                


                


                  
IPv4


                  

                    
0.0.0.0/8

                    
"This" network — the unspecified address block

                  

                  

                    
10.0.0.0/8

                    
Private — class A internal networks

                  

                  

                    
100.64.0.0/10

                    
Shared / CGNAT — catches cloud metadata endpoints (e.g. Alibaba 100.100.100.200)

                  

                  

                    
169.254.0.0/16

                    
Link-local — catches AWS metadata at 169.254.169.254

                  

                  

                    
172.16.0.0/12

                    
Private — class B internal networks

                  

                  

                    
192.168.0.0/16

                    
Private — class C home/office networks

                  


                  
IPv6


                  

                    
::

                    
Unspecified address

                  

                  

                    
fc00::/7

                    
Unique-local — private IPv6 equivalent of RFC 1918

                  

                  

                    
fe80::/10

                    
Link-local — auto-configured addresses

                  

                  

                    
::ffff:<blocked-v4>

                    
IPv4-mapped IPv6 — prevents hex bypass (e.g. ::ffff:a9fe:a9fe = 169.254.169.254)

                  

                

              


              
              

                

                  
                  Intentionally Allowed
                


                

                  

                    
127.0.0.0/8

                    
IPv4 loopback — local policy servers are a primary HTTP hook use case. Sidecar servers on localhost are explicitly supported.

                  

                  

                    
::1

                    
IPv6 loopback — same rationale as IPv4 loopback. Local servers reachable via [::1] are allowed.

                  

                


                

                  

                    
                    Guard bypass conditions
                  

                  

                    

                      
Sandbox

                      
When sandbox network proxy is active, the guard is skipped — the proxy enforces its own domain allowlist.

                    

                    

                      
HTTP Proxy

                      
When an env-var HTTP proxy is configured, the guard is bypassed (the proxy does DNS; validating the proxy IP is not useful).

                    

                  

                


                

                  
Implementation detail

                  
The guard uses a custom dns.lookup function passed to axios. This ensures the validated IP is the same one the socket connects to — closing the DNS rebind window that would exist if validation happened before the connection.
