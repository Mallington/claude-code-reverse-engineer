System

Peer agents with file-based mailboxes, three spawning backends, and a full team lifecycle. Internally codenamed “tengu.”

Architecture

Two Multi-Agent Models

Claude Code provides two distinct multi-agent architectures that serve different needs. Toggle to compare their design philosophies and trade-offs.

Swarm / Team Model
Codenamed “tengu” — TeamCreateTool + TeammateTool + SendMessageTool
Persistent team with named agents and a shared task list
File-based mailboxes in ~/.claude/teams/{name}/inboxes/
Agents poll at 500ms intervals; can message each other directly
Three spawning backends: tmux, iTerm2, in-process
Permission requests route through leader's terminal UI
One team per leader enforced; cleaned up on session end

Long-lived collaborative sessions with agent autonomy

Key Insight: The Swarm/Team model is designed for long-lived collaborative sessions where multiple agents work semi-independently on a shared task list. Coordinator mode is designed for structured workflows where one agent maintains full control of task decomposition and synthesis.

Permission Synchronization

Worker → Leader Permission Flow

When a teammate needs to perform an action requiring user approval, the permission must be routed to the leader who has the user's terminal. Two paths exist.

📁
Process-Based Path
tmux / iTerm2 backends — file-based
1
Worker encounters 'ask' permission on a tool action
2
Creates SwarmPermissionRequest with id perm-{ts}-{rand}
3
Written to permissions/pending/{id}.json inside team dir
4
Permission request sent to leader's mailbox via SendMessage
5
Leader shows approval dialog to user with tool-specific UI
6
Response written to worker's mailbox + resolved/{id}.json
7
Worker polls, reads response, proceeds or aborts
⚡
In-Process Shortcut
Direct ToolUseConfirmQueue bridge
1
Worker encounters 'ask' permission
2
Bash classifier pre-check — auto-approve safe commands first
3
getLeaderToolUseConfirmQueue() returns leader's UI queue reference
4
Worker pushes request directly onto queue with workerBadge (name + color)
5
User sees same tool UI as leader's own tools — BashPermissionRequest, FileEditToolDiff etc.
6
Permission updates written back to both worker context and leader's shared context
🔒
The In-Process Shortcut is elegant: the worker pushes its permission request directly onto the leader's UI queue. The user sees the exact same approval dialog they would for the leader's own tools, with a colored worker badge identifying which agent is asking. No filesystem round-trip needed.
Coordinator Mode

Four-Phase Coordinator Workflow

The coordinator system prompt defines a structured workflow with four phases. Click any phase to expand it, or use the controls to animate the full progression.

Phase 1
🔍
Research
Workers (parallel)
Investigate codebase, find relevant files
Read-only tasks run freely in parallel
Workers return <task-notification> with findings
Coordinator collects all results before moving on
Phase 2
🧠
Synthesis
Coordinator only
Coordinator reads all worker findings
Understands the problem space deeply
Writes specific implementation specs with file paths, line numbers, exact changes
Never lazy-delegates — must synthesize directly
Phase 3
⚙️
Implementation
Workers
Workers make targeted changes per spec
Write-heavy tasks serialized per file set
Self-contained prompts with all necessary context
Workers commit changes on completion
Phase 4
✅
Verification
Workers
Test that changes work as expected
Continue existing worker (high context overlap) or spawn fresh (low overlap)
Report pass/fail back to coordinator
Coordinator decides next action based on results
📋
The coordinator must synthesize. The prompt explicitly forbids lazy delegation like “based on your findings, fix the bug.” The coordinator must read worker findings, understand them, and write specific implementation specs with file paths, line numbers, and exact changes.
Lifecycle

Team Lifecycle Timeline

From TeamCreate to final cleanup. Teams are ephemeral by design — registered for session cleanup and removed when the session ends.

📄
Team Created
TeamCreateTool.call()
Leader calls TeamCreateTool. Team file written to ~/.claude/teams/{name}/config.json. One team per leader enforced — throws if leader already has a teamContext in AppState.
TeamFile: { name, leadAgentId, leadSessionId,
  teamAllowedPaths, members: [leader],
  createdAt }
🚀
Workers Spawned
TeammateTool → backend.spawn()
Each teammate is spawned via the selected backend. initializeTeammateHooks() configures the agent for team participation — reads team file, applies team-wide path permissions, registers Stop hook.
Member added to config.json • inbox file created
TEAMMATE_SYSTEM_PROMPT_ADDENDUM injected
buildInheritedCliFlags() + buildInheritedEnvVars()
📬
Workers Communicate
SendMessageTool → mailbox polling
Workers execute tasks, send status updates via SendMessageTool. Agents poll mailboxes every 500ms. Permission requests route through the leader. Task claiming from shared task list.
File locking: 10 retries, 5-100ms backoff
In-process: in-memory queue shortcut
Broadcast: to: "*" → all teammates
⏸️
Workers Idle
waitForNextPromptOrShutdown()
When a teammate finishes its work, the Stop hook sends an idle notification to the leader's mailbox with an accomplishment summary. The teammate enters waitForNextPromptOrShutdown(), polling at 500ms.
Polls: in-memory messages, mailbox,
available tasks, abort signal
setMemberActive(teamName, agentName, false)
🧹
Team Cleanup
registerTeamForSessionCleanup()
When the session ends, the team directory is removed. registerTeamForSessionCleanup() was added to fix gh-32730 — without it, team directories accumulated on disk indefinitely.
~/.claude/teams/{name}/ → deleted
Resolved permissions cleaned every 1 hour
