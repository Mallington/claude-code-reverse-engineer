Chapter 11 — Architecture Series

The Permission System

Every tool use is evaluated by a layered security pipeline — deny rules, mode checks, safety guards, and an optional AI classifier — before any action executes.

Section 01

The Seven Permission Modes

Five user-facing modes and two internal modes control the overall security stance. They determine whether actions are auto-allowed, auto-denied, or sent to the AI classifier.

🔒

default

Normal interactive mode. The user is prompted for approval on any tool use not covered by an explicit allow rule. This is the experience for most users.

Prompts user on unknown tools
✏️

acceptEdits

Auto-allows file edits (Write, Edit, NotebookEdit) when the target is within the working directory. Bash and other tools still require approval. Forms the auto-mode fast path.

Auto-allows in-CWD file edits
⚡

bypassPermissions

Allows all tool uses except those blocked by explicit deny rules, content-specific ask rules, or bypass-immune safety checks (.git/, .claude/, shell configs).

--dangerously-skip-permissions
🚫

dontAsk

Never prompts the user. Any tool use that would normally produce an ask result is converted to deny. Designed for non-interactive CI contexts.

ask → deny (no human present)
📋

plan

Planning mode. If bypassPermissions was active before entering plan mode (isBypassPermissionsModeAvailable), plan mode also bypasses. Otherwise behaves like default.

Inherits prior bypass state
🤖

auto

Internal

Uses an AI classifier (YOLO Classifier) instead of a human prompt. Only available when TRANSCRIPT_CLASSIFIER build feature is enabled. Feature-gated at compile time.

AI classifier replaces human
↑

bubble

Internal

Propagation mode used when permission decisions need to bubble up through nested contexts. Not directly settable by users or external configuration.

Permission propagation only

Section 02

The Decision Pipeline

hasPermissionsToUseToolInner() runs in three ordered phases. Deny and ask checks are evaluated first and cannot be overridden. Click any step to expand its details.

1 Deny & Ask Checks Cannot be overridden by any mode — fires first
1a Tool-level deny rule check → deny ›
getDenyRuleForTool() scans all deny rules from all sources (userSettings, projectSettings, cliArg, etc.). If any rule matches the tool and its input, the action is immediately denied. No mode can override this.
1b Tool-level ask rule check → ask ›
getAskRuleForTool() checks for explicit ask rules. If found, the user is prompted — unless a sandbox auto-allow applies. This is separate from the default "unknown tool" prompt.
1c Tool-specific permission check → varies ›
Each tool implements its own tool.checkPermissions(input, context). For Bash, this evaluates subcommand rules and command-level allow/deny matching. For file tools (Edit, Write), this checks path rules against allow/deny globs. Returns one of: allow, deny, ask, or passthrough.
1d Tool returned deny? → deny ›
If the tool's checkPermissions() returned deny (e.g., path is in a dangerous directory, or an explicit deny rule matched at the tool level), the action is blocked immediately.
1e Tool requires user interaction? → ask ›
If tool.requiresUserInteraction() is true and the current behavior is ask, the user must be prompted — even in bypassPermissions mode. Some tools inherently need human confirmation to function correctly.
1f Content-specific ask rule? → ask (bypass-immune) ›
If the decision reason is a rule with ruleBehavior === 'ask' (rather than a default passthrough), the ask is bypass-immune. Even bypassPermissions mode must prompt the user if an explicit ask rule matched the content.
1g Safety check for sensitive paths → ask (bypass-immune) ›
If the decision reason is a safetyCheck (sensitive path: .git/, .claude/, .vscode/, shell configs), the ask survives even bypassPermissions mode. These protections cannot be bypassed by any mode flag. Includes Windows UNC path blocking and cross-machine bridge messages.
2 Mode-Based Allow If no deny or ask fired, the mode decides
2a bypassPermissions (or plan with bypass available)? → allow ›
If the current mode is bypassPermissions, or if the mode is plan and isBypassPermissionsModeAvailable is true (the user was in bypass before entering plan mode), the action is allowed without prompting.
2b Tool-level allow rule? → allow ›
toolAlwaysAllowedRule() checks for an explicit allow rule matching this tool and input from any source. If found, the action proceeds without a user prompt. This is how permanent "always allow" preferences work.
3 Passthrough Conversion Deferred decisions become prompts
3 Tool returned passthrough? → ask ›
passthrough means the tool had no opinion. It is converted to ask so the user is always prompted. This prevents silent auto-allow on unknown inputs — a tool returning passthrough delegates to mode-based logic while still providing optional updatedInput suggestions.
⊕ Post-Processing (Outer Wrapper) Mode-specific transformations on the final result
P1 Result is allow → allow + reset counter ›
The consecutive denial counter (auto mode) is reset to zero, and the allow is returned.
P2 Result is ask — mode transformations → varies by mode ›
dontAsk mode: convert to deny.
auto mode: run AI classifier — non-classifier-approvable safety checks keep as ask/deny; acceptEdits fast path may allow; safe-tool allowlist may allow; otherwise run classifyYoloAction() → allow or deny.
shouldAvoidPermissionPrompts: run hooks, then auto-deny.

Key Insight

The passthrough behavior exists so a tool can signal "I have no opinion on this input" without forcing either allow or ask. A tool returning allow from checkPermissions() would bypass the mode check at step 2a — by returning passthrough, the tool delegates to mode-based logic while still providing suggestions.

Fail-Closed Design

When the classifier API is unavailable (network error, rate limit), the tengu_iron_gate_closed feature flag controls behavior. Default (true): deny the action and tell the model to retry or find an alternative approach. If false: fall through to normal permission handling (user prompt).

🔡
Case-Insensitive Normalization
All paths are lowercased before comparison via normalizeCaseForComparison(). Applied unconditionally (even on Linux) to handle macOS/Windows case-insensitive filesystems. Prevents .cLauDe/Settings.locaL.json bypass.
🧭
Path Traversal Detection
Rejects paths containing .. segments. Combined with symlink resolution for temp directories to prevent escaping the intended directory scope.
🪟
UNC Path Blocking
Blocks Windows UNC paths (\\server\share) which could be used to access network shares or bypass local path restrictions on Windows.
🎯
Skill Scoping
When editing files in .claude/skills/{skillName}/, the system offers a narrow permission scope — only that specific skill. Skill names containing .., glob metacharacters (* ? [ ]), or empty strings are rejected.

Why .claude/ is Protected

Claude Code's own config directory is treated as a dangerous directory — writable by neither bypassPermissions mode nor the default auto-mode flow. This prevents a compromised model from persisting new instructions or registering malicious MCP servers by modifying its own configuration files.

Section 06

Permission Rule Sources

Rules cascade from eight sources at runtime. User settings, project settings, policy, CLI args, and session responses are all merged into the ToolPermissionContext.

1
userSettings
~/.claude/settings.json — global per-user rules applied in every project.
2
projectSettings
.claude/settings.json — per-project, safe to check into the repo for sharing with teammates.
3
localSettings
.claude/settings.local.json — per-project, gitignored. For personal developer preferences that shouldn't be shared.
4
flagSettings
Feature flags and remote configuration pushed from Anthropic infrastructure.
5
policySettings
Organization-level policy, read-only. Allows enterprises to enforce mandatory deny rules across all users.
6
cliArg
Command-line --allowedTools and --disallowedTools flags. Useful for scripting and CI pipelines.
7
command
Custom slash commands that declare their required tool permissions inline.
8
session
Per-session rules accumulated when the user responds "always allow" or "always deny" to permission prompts during a session.

Rule Value Format

Rules use the format "ToolName" or "ToolName(content)". The content portion constrains the rule to specific inputs. Three matching strategies apply: exact (npm install), prefix (npm:* legacy syntax), and wildcard (git * compiles to /^git( .*)?$/s). The s (dotAll) flag is critical for heredoc commands that contain embedded newlines.
