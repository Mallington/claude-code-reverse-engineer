Reference

Message Type Catalog

Every line in a history.jsonl file is one of these six types. Each serves a distinct purpose in replaying and resuming a session.

metadata
Session start

Written once at the beginning of every session. Records the session identifier, model version, and creation timestamp — the envelope for the whole conversation.

{ "type": "metadata", "session_id": "sess_abc123", "model": "claude-sonnet-4-20250514", "created_at": "2026-03-23T17:12:16Z" }
user
Each user turn

Every message sent to the model. The content array can contain multiple parts — system reminders injected at turn start, skill catalog injections, and the actual user text are all separate items.

{ "type": "user", "role": "user", "content": [ {"type":"text","text":"<system-reminder>...</system-reminder>"}, {"type":"text","text":"Fix the failing tests"} ] }
assistant
Each model response

The model's reply. Content array mixes text blocks with tool_use blocks when the model calls tools. A single assistant turn can invoke multiple tools in sequence.

{ "type": "assistant", "role": "assistant", "content": [ {"type":"text","text":"I'll look at the tests..."}, {"type":"tool_use","id":"toolu_abc","name":"Bash"} ] }
tool_result
After each tool call

The output of a tool invocation, keyed by tool_use_id. Stored as a user-role message so the model can see it on the next turn. Large outputs may be stored separately in tool-results/.

{ "type": "tool_result", "role": "user", "content": [{ "type": "tool_result", "tool_use_id": "toolu_abc", "content": "Exit 1: 3 tests failed" }] }
progress
During tool execution

A display-only status update written while a tool is running. These lines drive the live UI indicators shown to the user. They are not sent back to the model — purely for UX feedback.

{ "type": "progress", "label": "Running Bash...", "tool_use_id": "toolu_abc" }
file_history_snapshot
Periodically

A point-in-time snapshot of relevant file contents and their mtimes. Allows Claude Code to detect whether files changed between sessions and show accurate diffs when resuming work.

{ "type": "file_history_snapshot", "files": [{ "path": "/src/auth.ts", "content": "...", "mtime": 1711217536 }] }
Nesting

Sub-Agent Storage

When the orchestrator spawns sub-agents, each gets its own history.jsonl in a nested directory. The on-disk tree mirrors the parent-child agent relationship.

sessions/sess-parent/ — Nested Agent Layout
sessions/sess-parent/ orchestrator
├─ history.jsonl parent conversation
└─ subagents/
├─ sess-implementer/ implementer agent
├─ history.jsonl implementer's work
└─ tool-results/
├─ toolu_read_123.json
└─ toolu_write_456.json
└─ sess-reviewer/ reviewer agent
├─ history.jsonl reviewer's work
└─ tool-results/ toolu_grep_789.json  +  toolu_read_012.json
history.jsonl — conversation JSONL
tool-results/ — persisted tool outputs
Each agent gets its own isolated session tree
Telemetry

Session Statistics

Claude Code accumulates token usage in stats-cache.json. Cache read tokens often dwarf input tokens — the context window is re-used aggressively across turns.

stats-cache.json
~/.claude/projects/<hash>/stats-cache.json

1.2M
New tokens sent to model each turn
Total Output Tokens
89K
Tokens generated by the model
Cache Read Tokens
987K
Tokens served from prompt cache — very cheap

12K
Tokens written to prompt cache
📈
Cache reads often exceed input tokens
The system prompt, tool definitions, and prior turns are stored in Anthropic's prompt cache. On each turn, those cached tokens are re-used at a fraction of the cost of sending them fresh. Over a long session, cache reads accumulate into the millions while actual new input stays modest.
