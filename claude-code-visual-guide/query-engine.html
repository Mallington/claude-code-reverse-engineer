StreamingToolExecutor

Tools start executing while the model is still streaming. Concurrent-safe tools (Read, Grep) run in parallel; non-concurrent tools (Bash, Write) queue for exclusive access.

Sibling Abort Controller: When a Bash tool errors, it calls siblingAbortController.abort('sibling_error') — killing all other running subprocesses immediately — without aborting the parent controller. Only Bash errors trigger sibling cancellation; Read/Grep/WebFetch are independent. Cancelled tools receive synthetic tool_result messages with is_error: true to satisfy the API's tool_use/tool_result pairing requirement.
Tool Lifecycle
Each tool moves through four states: queued → executing → completed → yielded. The executor calls canExecuteTool(isConcurrencySafe) before transitioning from queued to executing. A non-concurrent tool can only execute when no other tool is executing. A concurrent-safe tool can execute in parallel as long as all executing tools are also concurrent-safe.

Summary

Key Design Insights

Clean Separation
QueryEngine owns session state (messages, usage, permissions, transcript persistence) while query() owns the agentic loop. The query() generator knows nothing about SDK types or disk I/O.
Ordered Pipeline
Collapse runs before autocompact so it gets "first dibs" — if collapsing individual tool groups is sufficient, the more aggressive full autocompact is skipped. This preserves more granular context.
Error Withholding
Recoverable errors (413, max output, media size) are withheld during streaming, not surfaced. They are only revealed to callers if all recovery attempts fail — preventing SDK consumers from seeing transient errors.
Immutable State Transitions
Each recovery path writes a complete new State object with a transition field explaining why it continued. This makes the 8 distinct continue sites auditable and independently testable.
Streaming Overlap
StreamingToolExecutor starts concurrent-safe tools (Read, Grep) as soon as their tool_use block arrives in the stream — potentially completing before the model finishes streaming.
Transcript Before API Call
The transcript is written to disk before the API call begins. If the process is killed mid-response, --resume can still pick up the conversation. Cost: ~4ms I/O (skipped in --bare mode).
