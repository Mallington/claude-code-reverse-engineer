Lifecycle

Conversation Turn Structure

Each turn builds on the previous, adding tool results and system reminders, until compression fires near the limit.

Turn 1
System Prompt + 57 Tools + SystemReminder (full ~3K) + User Message → Response + ToolUse
Turn 2
System (HIT) + Tools + ToolResult + Reminder (light) + User Message → Response
Turn 3
System (HIT) + Tools + History + Reminder (light) + User Message → Response
... turns accumulate, history grows ...
Turn N
Context limit approaching → COMPRESSION: Summarize turns 1–N/2 → SessionStart re-fires → Continue with compressed history
Context Compression

How Compression Works

When conversation approaches the context limit, Claude compresses early history into a summary, preserving recent context and re-injecting bootstrap skills.

Prompt Caching

Cache Hit/Miss Pattern

Both system prompt blocks carry cache_control: ephemeral. System reminders exist specifically to keep the system prompt frozen — stable content means cache hits every turn after the first.

Cache read tokens are significantly cheaper than input tokens. Over a long session (20+ turns) this can cut costs by 40–60%.
📥
Cache Write Tokens
First time a block is seen. Full processing cost to write the cache entry. Happens only once per unique system prompt state.
⚡
Cache Read Tokens
Every subsequent turn. Much cheaper and faster than re-processing. The system prompt being frozen is what keeps these hits coming.
📄
Input Tokens
Non-cached content re-processed every turn — conversation history, tool results, user messages, and the light system reminder.
📤
Output Tokens
The model's generated response. Priced separately and typically a smaller fraction of the total token cost for tool-heavy workflows.
Optimizations

5 Token-Saving Techniques

These design choices collectively keep the effective per-turn token cost well below the naive baseline.

🔄
Deferred Tool Loading
MCP tool names are listed upfront but their full JSON schemas are not loaded until actually needed. A tool call to ToolSearch fetches the schema on demand.
Saves ~10,000–15,000 tokens/turn
🤖
Sub-Agent Context Isolation
Heavy research tasks are delegated to sub-agents via the RemoteTrigger tool. Only the summary result returns to the parent — not the entire research context.
Saves tens of thousands of tokens
🔍
Explore Agent for Research
Dedicated research sub-agents accumulate their own large context without polluting the parent session. The parent receives a concise answer only.
Parent context stays lean
📚
Skill On-Demand Loading
14 skills averaging ~1,500 tokens each — that's ~21,000 tokens if all pre-loaded. Instead, each skill is fetched individually only when relevant to the current task.
Saves up to ~21,000 tokens/turn
✂
Concise Response Guidelines
The system prompt explicitly enforces brevity — no filler, no summaries of what was done, no emoji. max_tokens: 32,000 limits response length. Less output = less context consumed.
Saves 500–3,000 tokens/response
Built-in Intelligence

Context Awareness

Claude Sonnet 4.6, 4.5, and Haiku 4.5 have built-in context awareness — the model actively tracks remaining space and adapts its behavior as the window fills.

🎯
Tracks Remaining Tokens
The model has visibility into how much context remains and factors this into every decision — what to load, what to delegate, how verbose to be.
📈
Adaptive Summarization
As context fills, earlier turns are summarized more aggressively. Tool results that are no longer relevant get dropped rather than carried forward.
🚫
Avoids Loading Unnecessary Skills
When the context window is tight, the model skips loading skill files it doesn't need — conserving space for the conversation itself.
🤝
Prefers Sub-Agent Delegation
When the parent context is nearly full, the model prefers to delegate research or file exploration to a sub-agent rather than expand its own context further.
