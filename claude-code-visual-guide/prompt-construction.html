System Prompt
Construction Pipeline

From named sections and memoization registries to cache-scoped TextBlockParam[] blocks — every stage of the assembly exposed.

How it works: Static sections before the SYSTEM_PROMPT_DYNAMIC_BOUNDARY sentinel are globally cached — identical across all users. Dynamic sections after it are per-session (null scope). The one VOLATILE section (mcp_instructions) recomputes every single turn, intentionally breaking the cache.

04 — Assembly Pipeline

The 8-Step Build Sequence

Complete call chain from source to API request, traced through the actual TypeScript source.

01
getSystemPrompt(tools, model, dirs, mcpClients)
src/constants/prompts.ts
The master builder. Evaluates build path (SIMPLE / PROACTIVE / Normal), builds section array, calls resolveSystemPromptSections() in parallel via Promise.all(). Inserts SYSTEM_PROMPT_DYNAMIC_BOUNDARY sentinel between static and dynamic halves when shouldUseGlobalCacheScope() is true.
→ string[] // array of assembled prompt sections
02
fetchSystemPromptParts()
src/utils/queryContext.ts
Calls three functions concurrently:
Parallel via Promise.all()
getSystemPrompt() getUserContext() → claudeMd + currentDate getSystemContext() → gitStatus
→ { defaultSystemPrompt, userContext, systemContext }
03
buildEffectiveSystemPrompt()
src/utils/systemPrompt.ts
Priority resolution — determines which system prompt actually gets used. Evaluates in order: override (loop mode) > coordinator > agent (custom/proactive appends vs replaces) > custom (--system-prompt flag) > default. Appends appendSystemPrompt at end unless override is set.
→ SystemPrompt // branded string[]
04
appendSystemContext()
src/utils/api.ts
Appends systemContext entries to the prompt array: gitStatus (current repo state) and cacheBreaker (if present). Git status is deliberately separate from the env_info_simple section — it's appended here, not baked into the memoization registry.
→ string[] // prompt array extended with git context
05
query() → callModel()
src/query.ts → src/services/api/claude.ts
Prepends to the system prompt array: getAttributionHeader(fingerprint) (billing metadata) and getCLISyspromptPrefix() (identity one-liner). Conditionally appends ADVISOR_TOOL_INSTRUCTIONS and CHROME_TOOL_SEARCH_INSTRUCTIONS if those features are active. Filters empty strings via .filter(Boolean).
→ string[] // final prompt array before block segmentation
06
prependUserContext()
src/utils/api.ts
Creates a synthetic user message (not system block) with isMeta: true flag, containing CLAUDE.md content and current date wrapped in <system-reminder> tags. Prepended at position 0 in the messages array — before all actual user messages. This is why CLAUDE.md is NOT in the system prompt.
→ messages[] // synthetic system-reminder prepended at index 0
07
buildSystemPromptBlocks()
src/services/api/claude.ts
Calls splitSysPromptPrefix() which segments the string[] into up to 4 cache-scoped blocks. Assigns cacheScope: 'global' to content before the boundary sentinel, null to content after it, and 'org' as fallback when no boundary is present.
→ TextBlockParam[] // { type:'text', text, cache_control: {type:'ephemeral', cacheScope} }
08
Final API Request
Anthropic Messages API
The completed request: system: TextBlockParam[], messages: Message[] (with system-reminder at index 0), tools (with cache_control on last element), plus model, betas, thinking config, etc.
→ POST /v1/messages // { system, messages, tools, model, … }
05 — Memoization

Section Cache Lifecycle

How systemPromptSection() and DANGEROUS_uncachedSystemPromptSection() manage the section cache — and why one section intentionally breaks it.

STATE.systemPromptSectionCache — session-scoped memoization registry
Factory A
systemPromptSection(name, compute)
Creates a memoized section. cacheBreak: false. Compute runs once; result stored in STATE.systemPromptSectionCache[name]. Subsequent turns return cached value without re-executing.
→
STATE.systemPromptSectionCache
{ "session_guidance": "…cached…", "memory": "…cached…", "env_info_simple": "…cached…", "language": "…cached…", "output_style": null, "scratchpad": "…cached…", "frc": "…cached…", }
✓ cache hit: return stored value
⚡
First Turn
Cache is empty. Each section's compute() function runs. Results stored in STATE.systemPromptSectionCache.
✓
Subsequent Turns
Cache has entries. resolveSystemPromptSections() returns cached values. No re-computation. Prompt stays frozen.
↺
/clear or /compact
clearSystemPromptSections() wipes the entire cache. Also resets beta header latches. Next turn recomputes all sections fresh.
DANGEROUS_uncachedSystemPromptSection
A second factory function creates volatile sections that recompute on every turn. The cacheBreak: true flag forces resolution to bypass the cache entirely. A mandatory reason parameter documents why cache-breaking is necessary. Currently, only one section uses this in normal operation:
DANGEROUS_uncachedSystemPromptSection(
  "mcp_instructions",
  () => computeMcpInstructions(mcpClients),
  "MCP servers connect/disconnect between turns"
)
Every time mcp_instructions changes (a server connects, disconnects, or reloads), the entire system prompt suffix after that point is invalidated. The API must re-tokenize and re-cache everything that follows. This is why connecting a new MCP server mid-session is intentionally expensive.

07 — Key Takeaways

What the Pipeline Reveals

Not a template — compiled

The system prompt is assembled from a named section registry with lazy evaluation, parallel resolution, and session-scoped memoization. Only one section is intentionally volatile.

Three tiers of cache scope

Global (identical for all users) → Org (shared within a team) → Null (per-session). The boundary sentinel enables infrastructure-level caching of static instruction text.

CLAUDE.md is not in the system prompt

It's injected as a synthetic user message with <system-reminder> tags via prependUserContext(). This keeps it entirely out of the system prompt cache.

MCP servers are expensive mid-session

Connecting a new MCP server invalidates the entire suffix cache. The DANGEROUS design is intentional — it documents exactly why this cost is accepted.
