Chapter 12 — The Memory System

The Memory System

Three independent systems — CLAUDE.md instructions, persistent auto-memory, and session memory — each solving a different continuity problem across different time horizons.

Section 03

Auto-Memory Types

Four types capture facts that are NOT derivable from the current project state. Click any card to see save rules and examples.

👤
user
User's role, goals, working style, knowledge level, and long-term preferences.
When to save
User states their role, domain expertise, long-term goals, communication preferences, or recurring workflows that define how they work.
Example memory file
--- name: user-role-observability type: user description: User is a data scientist focused on pipeline observability --- User is a data scientist. Primary focus: building observability tooling for ML training pipelines. Prefers Python, values explicit error handling.
Do NOT save
Code snippets, debugging history, file paths, or anything already captured in CLAUDE.md or the project itself.
🔥
feedback
Corrections AND confirmations of approach — both what to avoid and what has worked well.
When to save
User explicitly corrects Claude's behavior, requests a different approach, or confirms that a particular approach should be repeated in future.
Example memory file
--- name: database-testing-policy type: feedback description: Integration tests must use real DB, not mocks --- Integration tests must hit a real database. **Why:** Mock/prod divergence masked a broken migration in a prior incident. **How to apply:** When writing tests touching DB.
Do NOT save
One-off debugging solutions, ephemeral error fixes, or task-specific approaches that don't generalize to future sessions.
🎯
project
Ongoing work, goals, key decisions, deadlines, and project-specific context not in the codebase.
When to save
Important deadlines, strategic decisions, pending work that spans sessions, release constraints, or team context that affects how code should be written.
Example memory file
--- name: mobile-release-freeze type: project description: Merge freeze 2026-03-05 for mobile cut --- Merge freeze begins 2026-03-05 for mobile release. No schema migrations after 2026-03-01. Auth rewrite targeting this release window.
Do NOT save
Architecture, file structure, git history — all derivable from the repo. Also exclude ephemeral task details completed within a single session.
🔗
reference
Pointers to external systems, tools, trackers, dashboards, and resources relevant to this project.
When to save
User mentions external trackers, issue systems, dashboards, runbooks, or named processes that Claude should know about for future queries.
Example memory file
--- name: linear-ingest-project type: reference description: Pipeline bugs tracked in Linear INGEST --- Pipeline bugs are tracked in Linear project INGEST. Oncall runbook: internal wiki /runbooks/pipeline. Staging dashboard: grafana.internal/d/pipeline-stg
Do NOT save
API documentation or usage reference for tools Claude is actively using in the current session — the relevance engine filters these out automatically.

Section 05

Context Injection Flow

Each memory system injects its content at a different point in the message pipeline. Auto-memory appears in two places.

CLAUDE.md → system-reminder (first user message)
Auto-memory → system prompt section
Session memory → compaction summary
Section 06

Relevance Engine

Rather than loading all memory files, a Sonnet side-query scans headers and selects the top 5 most relevant for each query.

1
User query arrives
Main agent receives the user message. findRelevantMemories() is invoked before the response is generated.
2
Scan memory files
All *.md files in the memory directory are read (excluding MEMORY.md). Frontmatter is parsed for name, description, type. Capped at 200 files, sorted newest-first by mtime.
scanMemoryFiles(memoryDir) → MAX_MEMORY_FILES = 200 → sorted by mtime desc
3
Format manifest
One line per file: [type] filename (timestamp): description. This compact manifest is all Sonnet sees — no file contents.
[feedback] db-testing (2d ago): Integration tests must use real DB [project] auth-rewrite (5d ago): Auth rewrite in progress [user] user-role (12d ago): Data scientist, ML pipelines
4
Sonnet side-query
A small Sonnet query receives the user query, the manifest, and recently-used tools. It selects up to 5 files, spending max 256 tokens on the decision. Already-surfaced files are filtered from the budget.
{ selected_memories: ["db-testing.md", "auth-rewrite.md"] } // max 5 selected // recently-used tool docs excluded
5
Injected as attachments
Selected memory files are read from disk and injected into the model's context as file attachments alongside the user message. Only contents of selected files are loaded — not the entire memory directory.
Key Takeaways

What Makes This Architecture Tick

CLAUDE.md is not in the system prompt
It is injected as a <system-reminder> in a synthetic first user message via prependUserContext(). This affects caching: the system prompt is cached; the user-message injection is not. The isMeta: true flag prevents it from being shown to the user.
Extraction is a background forked agent
Both auto-memory extraction and session memory updates run as post-sampling hooks using runForkedAgent(). They share the parent's prompt cache. Tool permissions are tightly scoped: auto-memory gets read-anywhere plus write within its directory; session memory gets Edit-only on its single file.
projectSettings excluded from auto-memory path
The autoMemoryDirectory setting from project-level .claude/settings.json is intentionally excluded. A malicious repository could set it to ~/.ssh and gain silent write access. Only policy, local, and user settings are trusted for this path override.
