Prompt Caching

Cache Strategy

Both system blocks carry cache_control: { type: "ephemeral" }. Once warm, the entire system prompt is served from cache on every subsequent turn — bypassing re-tokenisation and re-processing.

Cache Hit / Miss Flow — per conversation turn Saves ~3,516 tokens re-processing per turn
Turn 1
System Prompt (~3,516 tok)
→
API
→
Cache MISS — processes & writes cache
Turn 2
System Prompt (~3,516 tok)
→
API
→
Cache HIT — reads cache ⚡
Turn 3
System Prompt (~3,516 tok)
→
API
→
Cache HIT — reads cache ⚡
⚡
After Turn 1, every subsequent turn saves ~3,516 tokens of re-processing. In a 20-turn session that's over 66,000 tokens avoided — a significant latency and cost reduction.

Frozen within a session. The system prompt never changes once the session starts — dynamic content (date, skills, CLAUDE.md) enters via system-reminder in the messages array instead.

Changes between sessions. A different working directory or a new day produces a slightly different <env> block, which requires a fresh Turn 1 cache miss — but each new session stays warm for its entire duration.

Why ephemeral? Ephemeral caches last for the lifetime of the API session rather than being persisted indefinitely, matching the expected usage pattern of a CLI tool that starts fresh each run.

Changing any system content breaks the cache. If you inject content into the system array mid-session — even one character — the cache fingerprint changes and Turn N becomes a miss. This is why the boundary between system and messages is architecturally significant.
