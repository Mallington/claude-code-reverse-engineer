Chapter 8 — Sub-Agent System

Sub-Agents

Context isolation through independent child sessions. Each sub-agent starts fresh, runs to completion, and returns its result to the parent.

Mechanics

How Sub-Agents Work

Every sub-agent is an independent Claude API session created by the parent. The Agent tool is the bridge — the parent calls it with a task prompt, and the result flows back as a tool_result.

1
New API Session

Parent creates a fresh POST /v1/messages call with its own system prompt and empty message history.

2
Filtered Tools

Sub-agent receives only a subset of tools based on its subagent_type — no skill catalog, no parent's tools list.

3
Runs to Completion

Sub-agent executes its task fully — reading files, writing code, calling APIs — then returns a structured result.

4
Result as tool_result

The sub-agent's final message is returned to the parent as a tool_result block in the parent's conversation.

Architecture

Context Isolation

The information barrier between parent and child is absolute. Sub-agents start fresh — they cannot see the parent's conversation history, system reminders, or skill catalog.

Parent Context
+
Full system prompt — user instructions, tool catalog, skill catalog
+
All tools — Agent, Edit, Write, Bash, MCP tools, etc.
+
Conversation history — all prior turns with the user
+
System reminders — injected context, memory, project state
+
Skill catalog — available skills from plugins
barrier
Sub-Agent Context
+
Sub-agent system prompt — condensed, task-focused, no user interaction instructions
+
Filtered tools only — subset based on subagent_type
+
Assigned prompt — the task description passed via Agent tool
-
No parent history — starts with an empty message log
-
No skill catalog — skills are NOT automatically available to sub-agents
-
No sibling awareness — cannot see other sub-agents' work
Context Management Benefits
Parent Stays Lean

Complex work delegated outward keeps the parent's context window focused on coordination.

Fresh Start Per Task

No context pollution between tasks — each sub-agent sees only its own work.

Parallel Work

Multiple tasks can run simultaneously in isolated sessions without interference.

Trade-off

Sub-agents have no memory of prior sub-agent work — parent must relay context explicitly.

Isolation Mode

Worktree Isolation

🌳
Git Worktree Isolation
When isolation: "worktree" is set in the Agent input, the sub-agent operates on its own independent copy of the repository.
Agent with worktree isolation
{
  "name": "Agent",
  "input": {
    "prompt": "Refactor authentication module",
    "subagent_type": "general-purpose",
    "isolation": "worktree"
  }
}
What It Is

A temporary Git worktree — a separate working directory linked to the same repository but on its own branch. The sub-agent gets its own complete copy of the repo state.

When to Use

Use for risky refactors, large-scale parallel work where agents might conflict, or when you want changes on a separate branch before merging back into the main working tree.

What Happens

A temp branch is created automatically. When the sub-agent finishes, changes live on that branch. If no meaningful changes were made, the worktree is auto-cleaned up.

Source Files

Agent Definitions

Plugin agents are defined as markdown files. Built-in agent types (general-purpose, Explore, Plan, claude-code-guide) are defined internally by Claude Code — their prompts are not exposed as files. Plugin agents like superpowers:code-reviewer ship as agents/code-reviewer.md inside the plugin directory.

Built-in Agent Types

The four built-in agent types are compiled into Claude Code and their system prompts are not exposed as files. Here is what is known about each from the documentation:

general-purpose
All tools available. A condensed version of the main system prompt focused on task execution rather than user interaction. No user interaction instructions, no skill catalog injection.
Explore
Read-only tools (Glob, Grep, Read, WebFetch, WebSearch). Tool set restricted at the system prompt level — no Edit, Write, Agent, or ExitPlanMode tools included in the tools array.
Plan
Same tool set as Explore (read-only). System prompt oriented toward architectural thinking, design tradeoffs, and producing implementation plans rather than executing code.
claude-code-guide
Glob, Grep, Read, WebFetch, WebSearch. Specialized system prompt for answering questions about Claude Code features, hooks, slash commands, MCP servers, settings, and IDE integrations.

runAgent() — The Core Execution Loop · runAgent.ts
1
Resolve model
getAgentModel(agent.model, mainLoopModel, override, permissionMode) — resolves "inherit", alias strings (haiku/sonnet), or a full model ID to a concrete model for the API call.
2
Create agentId
A unique identifier assigned to this agent run. Used for transcript storage keying and the /agents transcript viewer. May be supplied as an override (resume path).
3
Build initial messages
Combines forkContextMessages (filtered parent history, fork path only) with the incoming promptMessages. Fresh agents receive only the prompt — no parent conversation history.
4
Resolve user/system context
Calls getUserContext() / getSystemContext(). For read-only agents with omitClaudeMd: true, the CLAUDE.md hierarchy is skipped. Explore/Plan agents also have git status stripped — saves ~5–15 Gtok/week at fleet scale.
5
Configure permission mode
Decision point. Agent can override permission mode unless parent is bypassPermissions, acceptEdits, or auto. Async agents get shouldAvoidPermissionPrompts = true. Bubble mode surfaces prompts to the parent terminal.
6
Resolve tools
Decision point. Fork path: useExactTools = true → use availableTools verbatim (identical to parent for cache hit). Fresh path: resolveAgentTools(agentDef, availableTools, isAsync) applies all four filtering layers (global disallowed, custom extra, async allowlist, per-definition allow/deny).
7
Build system prompt
Decision point. Fork path: override.systemPrompt is passed directly (byte-identical to parent for prompt cache hit). Fresh path: getAgentSystemPrompt() → enhanceSystemPromptWithEnvDetails() produces an agent-specific condensed prompt.
8
Execute SubagentStart hooks
Runs any SubagentStart hooks registered for this agent. Hook output is collected as additional context injected before the query loop begins.
9
Register frontmatter hooks + preload skills
Frontmatter hooks are scoped to this agent's lifecycle and cleared on exit. Skills listed in the agent definition's frontmatter are resolved by name, loaded, and injected as user messages before the first API call.
10
Initialize agent-specific MCP servers
String refs (named servers) share an existing client. Inline definitions create a new MCP client that is cleaned up when the agent exits. The MCP connections are established here before tools are finalized.
11
Create subagent context
createSubagentContext(): sync agents share setAppState and the parent's AbortController. Async agents get a fully isolated context with a new AbortController so they can be cancelled independently.
12
Record initial messages to sidechain transcript
recordSidechainTranscript(agentId, ...) writes the startup state before the first query. This enables agent resume — if the run is interrupted, the transcript can be replayed by resumeAgent.ts.
13
THE QUERY LOOP — query()
for await (message of query({ messages, systemPrompt, userContext, systemContext, canUseTool, toolUseContext, querySource, maxTurns }))

This is the same query() function used by the main session. Sub-agents are not special — they run the identical engine. Inside the loop:
• API metrics (TTFT) forwarded to parent
• max_turns_reached signal → break
• Each message recorded to sidechain transcript
• Message yielded to caller (parent tool handler)

Thinking: disabled for regular sub-agents (thinkingConfig: { type: 'disabled' }) to control token costs. Fork children inherit parent's thinking config for cache-identical API prefixes.
14
Record each message (per turn)
Inside the query loop, every message yielded from query() is written to the sidechain transcript keyed by agentId. This powers the /agents command and enables mid-run resume.
15
Cleanup — finally block
Runs unconditionally (success, error, or abort):
• Clean up agent-specific MCP servers
• Clear session hooks registered in step 9
• Clean up prompt cache tracking
• Release file state cache
• Release perfetto tracing
• Remove agent's todo entries
• Kill any background bash tasks spawned by agent
