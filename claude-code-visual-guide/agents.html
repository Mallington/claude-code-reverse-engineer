Chapter 8 — Sub-Agent System

Sub-Agents

Context isolation through independent child sessions. Each sub-agent starts fresh, runs to completion, and returns its result to the parent.

Mechanics

How Sub-Agents Work

Every sub-agent is an independent Claude API session created by the parent. The Agent tool is the bridge — the parent calls it with a task prompt, and the result flows back as a tool_result.

1
New API Session

Parent creates a fresh POST /v1/messages call with its own system prompt and empty message history.

2
Filtered Tools

Sub-agent receives only a subset of tools based on its subagent_type — no skill catalog, no parent's tools list.

3
Runs to Completion

Sub-agent executes its task fully — reading files, writing code, calling APIs — then returns a structured result.

4
Result as tool_result

The sub-agent's final message is returned to the parent as a tool_result block in the parent's conversation.

Agent Tool Input 
{
  "name": "Agent",
  "input": {
    "description": "Fix failing tests",
    "prompt": "Full task description and all context the agent needs...",
    "subagent_type": "general-purpose",
    "model": "sonnet",
    "run_in_background": false,
    "isolation": "worktree"
  }
}

Architecture

Context Isolation

The information barrier between parent and child is absolute. Sub-agents start fresh — they cannot see the parent's conversation history, system reminders, or skill catalog.

Parent Context
+
Full system prompt — user instructions, tool catalog, skill catalog
+
All tools — Agent, Edit, Write, Bash, MCP tools, etc.
+
Conversation history — all prior turns with the user
+
System reminders — injected context, memory, project state
+
Skill catalog — available skills from plugins
barrier
Sub-Agent Context
+
Sub-agent system prompt — condensed, task-focused, no user interaction instructions
+
Filtered tools only — subset based on subagent_type
+
Assigned prompt — the task description passed via Agent tool
-
No parent history — starts with an empty message log
-
No skill catalog — skills are NOT automatically available to sub-agents
-
No sibling awareness — cannot see other sub-agents' work
Context Management Benefits
Parent Stays Lean

Complex work delegated outward keeps the parent's context window focused on coordination.

Fresh Start Per Task

No context pollution between tasks — each sub-agent sees only its own work.

Parallel Work

Multiple tasks can run simultaneously in isolated sessions without interference.

Trade-off

Sub-agents have no memory of prior sub-agent work — parent must relay context explicitly.

Isolation Mode

Worktree Isolation

🌳
Git Worktree Isolation
When isolation: "worktree" is set in the Agent input, the sub-agent operates on its own independent copy of the repository.
Agent with worktree isolation
{
  "name": "Agent",
  "input": {
    "prompt": "Refactor authentication module",
    "subagent_type": "general-purpose",
    "isolation": "worktree"
  }
}
What It Is

A temporary Git worktree — a separate working directory linked to the same repository but on its own branch. The sub-agent gets its own complete copy of the repo state.

When to Use

Use for risky refactors, large-scale parallel work where agents might conflict, or when you want changes on a separate branch before merging back into the main working tree.

What Happens

A temp branch is created automatically. When the sub-agent finishes, changes live on that branch. If no meaningful changes were made, the worktree is auto-cleaned up.

Source Files

Agent Definitions

Plugin agents are defined as markdown files. Built-in agent types (general-purpose, Explore, Plan, claude-code-guide) are defined internally by Claude Code — their prompts are not exposed as files. Plugin agents like superpowers:code-reviewer ship as agents/code-reviewer.md inside the plugin directory.

superpowers:code-reviewer ~/.claude/plugins/cache/claude-plugins-official/superpowers/5.0.5/agents/code-reviewer.md
markdown

Built-in Agent Types

The four built-in agent types are compiled into Claude Code and their system prompts are not exposed as files. Here is what is known about each from the documentation:

general-purpose
All tools available. A condensed version of the main system prompt focused on task execution rather than user interaction. No user interaction instructions, no skill catalog injection.
Explore
Read-only tools (Glob, Grep, Read, WebFetch, WebSearch). Tool set restricted at the system prompt level — no Edit, Write, Agent, or ExitPlanMode tools included in the tools array.
Plan
Same tool set as Explore (read-only). System prompt oriented toward architectural thinking, design tradeoffs, and producing implementation plans rather than executing code.
claude-code-guide
Glob, Grep, Read, WebFetch, WebSearch. Specialized system prompt for answering questions about Claude Code features, hooks, slash commands, MCP servers, settings, and IDE integrations.
