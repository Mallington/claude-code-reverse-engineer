Phase 3 Deep Dive

SSE Streaming — How Bytes Become Responses

Every API response streams back as Server-Sent Events over a single HTTP connection. Here is the exact event sequence from a real captured session.

The HTTP Response
HTTP/1.1 200 OK
Content-Type: text/event-stream; charset=utf-8
The connection stays open. Events arrive one by one as the model generates tokens.
SSE Wire Format
event: content_block_delta
data: {"type":"content_block_delta", ...}
⏎ (blank line = event boundary)
Each event has a type line and a JSON data line, separated by blank lines.

SSE Event Types — The Complete Sequence

Order Event Count What it carries
1 message_start 1 Model ID, message ID, role, initial usage (input tokens, cache stats)
2 content_block_start 1 per block Block index + type ("text" or "tool_use" with tool name & ID)
— ping periodic Keep-alive heartbeat — no payload, keeps connection open
3 content_block_delta many The actual content — text_delta (text chunks) or input_json_delta (tool input JSON chunks)
4 content_block_stop 1 per block Block complete — signals end of one content block
5 message_delta 1 Final usage stats (output tokens) + stop_reason ("end_turn" or "tool_use")
6 message_stop 1 Connection complete — stream ends

Pseudo-Smooth Typing — The Token Buffer

Claude's SSE chunks arrive in large bursts (17–155 characters per delta), but the chat UI makes text appear to flow character by character. This is achieved through a token buffer with a drip-feed timer — a technique we confirmed by instrumenting the claude.ai DOM with a MutationObserver.

The Algorithm
buffer = ""
onSSEDelta(chunk):
buffer += chunk
drainLoop // runs every ~6ms:
if buffer.length > 0:
render buffer[0] to DOM
buffer = buffer.slice(1)
Observed Behavior (claude.ai)
~6ms/char Drain rate — 1 character rendered per tick
~90ms Typical gap between SSE deltas
~60 chars Average delta chunk size
Buffer grows When model outputs faster than drain rate (60 chars in 90ms vs drain at 1 char/6ms = 15 chars drained)
Buffer Race Condition
t=0ms SSE delta arrives: "Hello! I'm Claude" (17 chars → buffer)
t=6ms drain: Hello! I'm Claude (buffer: 16 remaining)
t=12ms drain: Hello! I'm Claude (buffer: 15)
t=... drain continues at 6ms/char...
t=90ms NEXT SSE delta: ", an AI assistant built..." (155 chars → buffer!)
only ~15 chars drained so far → buffer now has 155 + 2 = 157 chars
t=180ms NEXT SSE delta: " are the tools..." (70 chars → buffer grows more)
buffer accumulates faster than drain → visible catch-up bursts

Why "pseudo-smooth"? The effect looks like the model is thinking and typing in real time, but it's an illusion. The model has already generated the tokens — they're sitting in a client-side buffer waiting to be rendered. Toggle the Pseudo-smooth typing switch above to see the difference: with it off, you see the raw SSE chunk boundaries; with it on, you see the smoothed drip-feed that the claude.ai UI presents.

Key Insight

The stop_reason in message_delta determines what happens next. If it's "end_turn", Claude Code shows the response and waits for user input. If it's "tool_use", Claude Code executes the requested tool, appends a tool_result message, and makes another API call — the agentic loop continues.

A single user message can trigger dozens of API round-trips. Each round-trip is a full SSE stream with its own message_start → deltas → message_stop sequence. The conversation grows with each loop until the model finally returns end_turn.

Architecture Insights

Key Design Decisions

Three architectural choices that explain why the request is structured the way it is.

Split System Prompt

The system prompt is split into two blocks: a short identity string and long instructions. Both are marked cache_control: ephemeral, so Anthropic's prompt cache stores them between calls — saving latency and cost.

Block 0: 62 chars (identity)
Block 1: ~13,854 chars (instructions)
Total: ~3,500 tokens cached
Dynamic Context in Messages

The current date, working directory, and other volatile context go into the messages array, not the system prompt. This keeps the system prompt identical across calls, preserving the prompt cache hit.

system[] — frozen, cached
messages[0].content[0] — <system-reminder>
messages[0].content[1] — user text
Streaming SSE

The response streams as Server-Sent Events, not a single JSON blob. This lets Claude Code display output incrementally and detect a tool_use block mid-stream to start tool execution immediately.

stream: true in request
?beta=true — extended context + tool improvements
tool_use detected → execute → loop
